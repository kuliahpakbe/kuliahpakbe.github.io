Manajemen Paket & Layanan Server

Jadilah penguasa ekosistem Linux. Pelajari cara menginstal perangkat lunak dari repositori global, mengendalikan layanan latar belakang (daemon), dan melacak setiap nafas sistem operasi melalui berkas log.

Buka Simulator Layanan

Manajemen Paket (Package Management)

Di Linux, Anda tidak mencari aplikasi berekstensi .exe di internet lalu menginstalnya secara manual. Kita menggunakan sistem Repository yang aman dan terpusat.

Advanced Package Tool (APT)

Keluarga Debian/Ubuntu menggunakan APT untuk mengelola perangkat lunak. APT secara cerdas mengunduh aplikasi beserta seluruh pustaka pendukungnya (dependencies) secara otomatis.

3 Perintah Suci Administrator:

  • sudo apt update Mengambil daftar katalog aplikasi terbaru dari server pusat (Tidak menginstal apapun).
  • sudo apt upgrade Memperbarui (update) seluruh aplikasi yang ada di komputer ke versi terbaru berdasarkan katalog.
  • sudo apt install nginx Mengunduh dan memasang aplikasi tertentu (misal: Nginx Web Server).

Manajemen Layanan (Systemd)

Setelah aplikasi server (seperti Nginx atau MySQL) diinstal, mereka harus berjalan terus-menerus di latar belakang tanpa henti. Proses ini disebut Daemon.

Apa itu Systemd?

Systemd (dengan huruf 'd' berarti daemon) adalah manajer sistem dan layanan untuk OS Linux modern. Ia adalah proses induk (PID 1) yang bertugas menyalakan, mematikan, dan memonitor aplikasi latar belakang lainnya.

Perintah systemctl

Kita berinteraksi dengan systemd menggunakan alat CLI bernama systemctl. Alat ini digunakan untuk menyuruh layanan agar start, stop, atau restart (misal jika kita mengubah file konfigurasi).

Enable vs Start

start hanya menyalakan layanan saat ini saja. Jika server direstart, layanan akan mati.
enable mendaftarkan layanan tersebut agar hidup otomatis setiap kali server baru di-booting (Auto-Start).

Analisis Log & Keamanan

Memahami service saja tidak cukup. Administrator yang andal harus tahu cara melacak kesalahan ketika sebuah layanan menolak untuk hidup (Failed State).

Jantung Diagnostik: journalctl

Di era systemd, seluruh log sistem, kernel, dan aplikasi digabung ke dalam satu basis data biner terpusat yang disebut journal. Untuk membacanya, kita tidak lagi membuka /var/log/syslog secara manual dengan editor teks, melainkan menggunakan perintah journalctl.

# Melihat log khusus layanan Nginx secara live (terus memantau)
journalctl -u nginx -f
# Melihat log sistem yang error (kritis) sejak server hidup
journalctl -b -p err
🔥

Troubleshooting: Port Collision

Kasus Industri yang Sering Terjadi: Anda mencoba menjalankan systemctl start nginx, namun statusnya merah FAILED. Mengapa?

Jika Anda mengecek log dengan journalctl -u nginx, Anda biasanya akan menemukan pesan "bind() to 0.0.0.0:80 failed (98: Address already in use)". Ini berarti ada aplikasi lain (misal: Apache) yang sudah lebih dulu memakai Port 80 (port standar web). Dua layanan berbeda tidak bisa mendengarkan pada port yang sama persis.
